## 1. 为什么发生本轮纠偏

此前开发按数据库、PDF、研究计划和绘图模块连续扩张，但比赛基础功能、真实语料、自然多轮和
用户验证没有同步闭环。最新提交还删除了 `SOLUTION_DESIGN.md` 等关键文档，并在 `.gitignore`
加入 `*.md`，导致新的项目约束无法被 Git 跟踪。

本轮已经重新以正式赛题 `XH-202620` 为基线。不得将“规则允许使用其他工具”误解为项目可以弱化讯飞平台。
项目已审核冻结正式比赛主链路为：

`Vue → FastAPI → 星辰 Agent → 星火/科技文献模型、MaaS → 自研结构化工具`

Vue 继续作为正式 Demo 和证据界面，FastAPI 是受控工具网关。这不代表星辰是官方唯一允许的入口，而是本项目的参赛技术策略。

### 不能宣称已完成

- 星辰或星火线上真实调用：没有真实联调证据。
- MaaS“TAD实验协议风险诊断”：没有数据契约、冻结盲测集、ServiceID和前后对比。
- 目标技术栈中的 Element Plus、Pinia、Celery、Redis、Cytoscape.js 尚未在当前代码实现。
- 目标Milvus 2.6.x混合检索未实现；当前依赖范围仍是`pymilvus>=2.5,<3`。
- 比赛级权威知识库：当前 5 篇清单中只有 1 篇 `development_seed`，0 篇双审/冻结。
- 比赛级 RAG：默认检索后端是 `demo`，嵌入是 `hash`，Milvus 尚未形成真实检索闭环。
- 多论文比较和研究机会：真实语料不足时正确返回 `insufficient_evidence`。
- 自然多轮智能体：默认是可测试的关键词/规则导航，不是大模型自主规划。
- 真实用户效果：仓库没有至少 2 名用户记录，也没有 3 个正式问题报告。
- 个性化：没有基于用户水平和历史的可验证自适应策略。
